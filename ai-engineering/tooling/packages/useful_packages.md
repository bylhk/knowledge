# Useful AI/LLM Packages

**AI Framework**
| Tool | License | Stars | Purpose |
|------|---------|-------|---------|
| [LangChain](https://github.com/langchain-ai/langchain) | MIT | ~100k | The most widely adopted LLM orchestration framework with abstractions for chains, retrievers, tools, and memory. Huge ecosystem of integrations (700+ including every major vector DB, LLM provider, and tool). Can be over-abstracted for simple use cases but unmatched for breadth. |
| [LlamaIndex](https://github.com/run-llama/llama_index) | MIT | ~40k | Data framework purpose-built for connecting LLMs to external data sources — excels at indexing, retrieval, and query engines over documents. Provides high-level abstractions like `VectorStoreIndex` and `KnowledgeGraphIndex` for rapid RAG setup. Better suited than LangChain when the primary task is structured data retrieval. |
| [AutoGen](https://github.com/microsoft/autogen) | CC-BY-4.0 | ~40k | Microsoft's multi-agent conversation framework where agents with different roles collaborate via message passing. Supports human-in-the-loop, code execution, and group chat patterns. Good for complex tasks requiring planning, debate, or specialised agent delegation. |
| [CrewAI](https://github.com/crewAIInc/crewAI) | MIT | ~25k | Role-based multi-agent framework where you define agents with backstories, goals, and tools that collaborate on tasks. Simpler API than AutoGen with built-in sequential/hierarchical process types. Good for assembling "teams" of specialised agents for content generation or research. |
| [Semantic Kernel](https://github.com/microsoft/semantic-kernel) | MIT | ~22k | Microsoft's SDK for building AI agents with a plugin architecture — strong C#/Java/Python support and enterprise patterns. Native integration with Azure OpenAI, Copilot stack, and Microsoft 365. Best choice for .NET shops or teams already in the Microsoft ecosystem. |
| [Haystack](https://github.com/deepset-ai/haystack) | Apache 2.0 | ~18k | Pipeline-based NLP framework with first-class support for RAG, document processing, and model serving. Clean component abstraction (Retrievers, Readers, Generators) that's more opinionated than LangChain. Strong production focus with pipeline serialisation and REST API deployment. |
| [smolagents](https://github.com/huggingface/smolagents) | Apache 2.0 | ~15k | Hugging Face's minimal agent framework — agents write and execute Python code as their action space rather than JSON tool calls. Very lightweight (~1000 lines core) with support for multi-agent hierarchies. Good for code-generation agents and research experimentation. |
| [LangGraph](https://github.com/langchain-ai/langgraph) | MIT | ~10k | Extends LangChain with cyclical graph execution for building stateful, multi-step agents with human-in-the-loop support. Models agent logic as nodes and edges with persistent state (checkpointing, time-travel). The go-to for complex agent workflows that need branching, loops, and recovery. |
| [MCP SDK](https://github.com/modelcontextprotocol/python-sdk) | MIT | ~5k | Anthropic's open standard for connecting LLMs to tools and data sources via a client-server protocol. Provides a universal interface so tools are written once and work with any MCP-compatible host (Claude, Kiro, etc.). Growing ecosystem of community servers for databases, APIs, and file systems. |

**Prompt Optimisation**
| Tool | License | Stars | Purpose |
|------|---------|-------|---------|
| [DSPy](https://github.com/stanfordnlp/dspy) | MIT | ~34k | Replaces hand-written prompts with declarative "signatures" that are automatically compiled into optimised prompts/few-shot examples via teleprompters. Supports chain-of-thought, RAG, and multi-hop reasoning patterns as composable modules. Evaluates and optimises end-to-end on your metrics, not just individual prompt quality. |
| [TextGrad](https://github.com/zou-group/textgrad) | MIT | ~2k | Applies backpropagation-style optimisation to text — uses LLM feedback as "gradients" to iteratively improve prompts, code, or solutions. Enables compound AI systems where each component is optimised via natural language feedback. Research-oriented but effective for complex prompt chains. |
| [OPRO](https://github.com/google-deepmind/opro) | Apache 2.0 | ~2k | Uses an LLM as the optimiser itself — iteratively generates and scores prompt candidates based on a training set. Simple "meta-prompt" approach from DeepMind that requires no gradient computation. Effective for instruction optimisation but slower due to many LLM calls per iteration. |
| [EvoPrompt](https://github.com/beeevita/EvoPrompt) | MIT | ~1k | Applies evolutionary algorithms (genetic algorithms, differential evolution) to prompt optimisation. Maintains a population of prompt variants and evolves them through crossover and mutation using an LLM. Useful when the search space is large and gradient-free exploration is needed. |

**Fine-Tuning**
| Tool | License | Stars | Purpose |
|------|---------|-------|---------|
| [LLaMA-Factory](https://github.com/hiyouga/LLaMA-Factory) | Apache 2.0 | ~40k | All-in-one fine-tuning platform with web UI supporting 100+ models, LoRA/QLoRA/GLoRA/full fine-tuning, and DPO/RLHF/PPO alignment. Zero-code training via YAML config or browser interface — just point at your dataset and go. Best starting point for fine-tuning if you don't want to write training loops. |
| [Unsloth](https://github.com/unslothai/unsloth) | Apache 2.0 | ~25k | Optimised LoRA/QLoRA training that's 2-5x faster and uses 60-80% less memory than standard Hugging Face training through custom CUDA kernels. Drop-in replacement for Hugging Face `Trainer` — just wrap your model with `FastLanguageModel`. Enables fine-tuning 70B models on a single GPU that would normally require multi-GPU setups. |
| [MLX](https://github.com/ml-explore/mlx) | MIT | ~18k | Apple's ML framework optimised for Apple Silicon (M1/M2/M3/M4) with unified memory and lazy evaluation. Supports LoRA fine-tuning and inference of LLMs natively on Mac without external GPU. Growing ecosystem with mlx-lm for model conversion and training scripts. |
| [peft](https://github.com/huggingface/peft) | Apache 2.0 | ~17k | Hugging Face's library implementing parameter-efficient fine-tuning methods — LoRA, QLoRA, IA3, prefix tuning, prompt tuning, and adapters. The foundational building block used by Unsloth, Axolotl, and LLaMA-Factory under the hood. Best when you want fine-grained control over adapter configuration and composition. |
| [bitsandbytes](https://github.com/TimDettmers/bitsandbytes) | MIT | ~14k | CUDA library for 4-bit and 8-bit quantisation enabling QLoRA — load a 70B model in 4-bit (35GB) and train LoRA adapters on top. Provides `Linear4bit` and `Linear8bitLt` layers that are drop-in replacements for `nn.Linear`. Essential dependency for memory-efficient fine-tuning on consumer GPUs. |
| [trl](https://github.com/huggingface/trl) | Apache 2.0 | ~11k | Hugging Face's training library for RLHF, DPO, SFT, and reward modelling with `Trainer`-compatible API. Provides `SFTTrainer`, `DPOTrainer`, and `PPOTrainer` that handle the complex training loops for alignment. The standard for preference-based fine-tuning when you have human feedback data. |
| [Axolotl](https://github.com/OpenAccess-AI-Collective/axolotl) | Apache 2.0 | ~8k | YAML-config-driven fine-tuning supporting multi-GPU, DeepSpeed, LoRA/QLoRA, and complex dataset mixing in a single config file. Handles data preprocessing, conversation templates, and evaluation automatically. Preferred by the open-source fine-tuning community for reproducible training runs. |
| [torchtune](https://github.com/pytorch/torchtune) | BSD-3 | ~5k | PyTorch's official LLM fine-tuning library — native, hackable, and dependency-light with recipes for LoRA, QLoRA, full fine-tuning, and DPO. Designed to be readable and modifiable (no deep abstraction layers) for researchers who want to understand every step. Supports Llama, Mistral, Gemma, and Phi model families. |
| [DeepSpeed](https://github.com/microsoft/DeepSpeed) | Apache 2.0 | ~36k | Microsoft's distributed training library with ZeRO optimiser stages (1/2/3) that shard model states across GPUs for massive model training. Enables training of 100B+ parameter models across multiple nodes with near-linear scaling. Also provides inference optimisations (DeepSpeed-Inference) for serving large models. |

**RAG System**
| Tool | License | Stars | Purpose |
|------|---------|-------|---------|

| [ChromaDB](https://github.com/chroma-core/chroma) | Apache 2.0 | ~16k | Embedded vector database designed for simplicity — `pip install chromadb` and you're running locally with persistent storage. | [pgvector](https://github.com/pgvector/pgvector) | PostgreSQL | ~13k | PostgreSQL extension adding vector similarity search (cosine, L2, inner product) with HNSW and IVFFlat indexes. Keep vectors alongside your relational data without a separate vector DB — great for existing Postgres deployments. Supports exact and approximate search with filtering on regular SQL columns. |
| [FlagEmbedding](https://github.com/FlagOpen/FlagEmbedding) | MIT | ~8k | BAAI's embedding toolkit containing the BGE model family (top MTEB leaderboard) with support for instruction-tuned embeddings and MRL. Includes training scripts for custom embedding models and long-context retrieval (LLM-Embedder). The BGE-M3 model supports dense, sparse, and ColBERT retrieval in a single model. |
| [LanceDB](https://github.com/lancedb/lancedb) | Apache 2.0 | ~5k | Serverless embedded vector DB built on Lance columnar format — no server process needed, just a library. Supports multi-modal data (text, images, video) with automatic versioning and zero-copy access. Good for applications needing vector search without infrastructure complexity. |
| [ColBERT](https://github.com/stanford-futuredata/ColBERT) | MIT | ~3k | Late-interaction retrieval model that represents queries and documents as sets of token embeddings, enabling fine-grained matching. Achieves near-cross-encoder quality at retrieval speed via pre-computed document token embeddings (PLAID engine). The theoretical foundation behind RAGatouille and many production re-ranking systems. |
| [RAGatouille](https://github.com/bclavie/RAGatouille) | Apache 2.0 | ~3k | Makes ColBERTv2 accessible for RAG with a simple API — `RAGPretrainedModel.from_pretrained()` then `.search()`. Handles index creation, compression, and querying without needing to understand late-interaction internals. Drop-in upgrade from bi-encoder retrieval with significant relevance improvements. |
| [FlashRANK](https://github.com/PrithivirajDamodaran/FlashRank) | Apache 2.0 | ~2k | Ultra-lightweight cross-encoder re-ranker (~4MB model) that runs on CPU in milliseconds — no GPU required. Provides a simple `Ranker().rerank()` API for re-scoring retrieved passages. Best for adding re-ranking to resource-constrained environments. |
| [rerankers](https://github.com/AnswerDotAI/rerankers) | Apache 2.0 | ~1k | Unified interface for all re-ranking approaches — cross-encoders, ColBERT, LLM-based, and API rankers through one API. Swap between `FlashRankRanker`, `TransformerRanker`, `CohereRanker` with no code changes. Created by the Answer.AI team for production RAG flexibility. |
| [rank_bm25](https://github.com/dorianbrown/rank_bm25) | Apache 2.0 | ~1k | Pure Python BM25 implementation for sparse keyword retrieval — the classic baseline for information retrieval. Use alongside dense vectors for hybrid search (BM25 handles exact keyword matches that embeddings miss). Lightweight with no dependencies beyond numpy. |
| [Pinecone](https://www.pinecone.io) | Proprietary | — | Fully managed vector database with serverless pricing, hybrid search, and namespace-based multi-tenancy. Zero infrastructure management with automatic scaling and 99.99% SLA. Industry standard for teams that want managed vector search without ops burden. |
| [Cohere Rerank](https://cohere.com/rerank) | Proprietary | — | Production-grade neural re-ranker API that significantly improves retrieval quality with a single API call. State-of-the-art cross-encoder performance without self-hosting GPU infrastructure. De facto standard for teams needing high-quality re-ranking without model management. |

|------|---------|-------|---------|

| [DeepSpeed](https://github.com/microsoft/DeepSpeed) | Apache 2.0 | ~36k | Also provides inference optimisations (DeepSpeed-Inference) with tensor parallelism, kernel fusion, and quantisation for serving large models. | [Py-SPY](https://github.com/benfred/py-spy) | MIT | ~13k | Sampling profiler for Python that attaches to running processes without code modification or restart. | [Scalene](https://github.com/plasma-umass/scalene) | Apache 2.0 | ~12k | AI-powered Python profiler that separately measures CPU, GPU, and memory usage line-by-line with copy volume tracking. Automatically suggests optimisations using an LLM. Best for identifying memory leaks and GPU underutilisation in ML pipelines. |
| [TensorRT-LLM](https://github.com/NVIDIA/TensorRT-LLM) | Apache 2.0 | ~10k | NVIDIA's optimised LLM inference library with kernel fusion, quantisation (FP8, INT4/INT8), and in-flight batching for maximum GPU utilisation. Requires NVIDIA GPUs but delivers the highest possible single-GPU throughput. Best for production deployments on NVIDIA hardware where latency matters. |
| [SGLang](https://github.com/sgl-project/sglang) | Apache 2.0 | ~8k | Fast LLM serving runtime with RadixAttention for KV cache reuse across requests sharing common prefixes. Provides a frontend language for programming complex LLM calls (multi-turn, branching, constrained decoding). Competitive with vLLM on throughput with better support for structured generation. |
| [GPTCache](https://github.com/zilliztech/GPTCache) | MIT | ~7k | Semantic caching layer for LLM APIs — caches responses and returns them for semantically similar (not just identical) queries. Reduces API costs and latency by 10-100x for repeated/similar questions. Pluggable with different similarity backends (FAISS, ChromaDB, Redis). |
| [FlashAttention](https://github.com/Dao-AILab/flash-attention) | BSD-3 | ~7k | IO-aware exact attention algorithm that's 2-4x faster and uses 5-20x less memory than standard attention by tiling and recomputation. Now integrated into PyTorch 2.0+ (`torch.nn.functional.scaled_dot_product_attention`) and most model implementations. The foundational optimisation enabling long-context models (128k+ tokens). |
| [ExLlamaV2](https://github.com/turboderp/exllamav2) | MIT | ~4k | Highly optimised GPTQ/EXL2 inference engine for NVIDIA GPUs — fastest option for quantised model inference on consumer hardware. Supports dynamic quantisation mixing (different bits per layer) for optimal quality/speed tradeoff. Popular in the local LLM community for running 70B models on single GPUs. |
| [AWQ](https://github.com/mit-han-lab/llm-awq) | MIT | ~3k | Activation-Aware Weight Quantisation that identifies salient weights (1% of weights handle most activations) and protects them during 4-bit quantisation. Better quality than naive GPTQ at the same bit-width, especially for smaller models. Supported natively in vLLM, TensorRT-LLM, and transformers. |
| [DeepSpeed-MII](https://github.com/microsoft/DeepSpeed-MII) | Apache 2.0 | ~2k | Microsoft's model inference library providing low-latency serving with DeepSpeed's optimisations via a simple API. Supports load balancing, replica management, and non-persistent deployment modes. Good for Azure/Microsoft ecosystem deployments but less community adoption than vLLM. |
